<task type="auto">
  <name>Task 2: Implement fetch pipeline — HTTP retry, cache with TTL, SHA256 verification</name>
  <files>fust/src/fetch.rs</files>
  <action>
Implement all functions in `fust/src/fetch.rs`. Follow modules.sh logic exactly:

**`FetchConfig::from_env()`:**
- `base_url`: `std::env::var("FLU_MODULES_BASE_URL")` or `DEFAULT_BASE_URL`
- `cache_dir`: `std::env::var("FLU_CACHE_DIR")` or `XDG_CACHE_HOME/flu.sh` or `HOME/.cache/flu.sh`
  - Use `std::env::var("XDG_CACHE_HOME")` first, then fall back to `dirs::home_dir()` + `/.cache`
  - Since we don't have the `dirs` crate, use `std::env::var("HOME")` + `/.cache/flu.sh` as fallback
- `cache_ttl`: `std::env::var("FLU_CACHE_TTL")` parsed as u64, or `DEFAULT_CACHE_TTL`

**`resolve_url(config, action_id)`:**
- Return `format!("{}{}.sh", config.base_url, action_id)`
- No community/* handling (Phase 19 scope)

**`fetch_manifest(config)`:**
- GET `{base_url}MANIFEST.sha256` using `reqwest::blocking::get`
- Connect timeout: 5 seconds (matching modules.sh `--connect-timeout 5`)
- On any error: print warning to stderr, return `None` (soft-fail per modules.sh D-03)
- On success: return `Some(body_text)`

**`fetch_module(config, action_id)`** — the main pipeline:

1. **Cache check** (matching modules.sh lines 115-127):
   - Cache file path: `{cache_dir}/{action_id}` (action_id is e.g. "install_go")
   - If file exists and is non-empty:
     - Get file modification time via `std::fs::metadata().modified()`
     - Compute age: `now - mtime` in seconds
     - If `age < cache_ttl`: print `[cached] {action_id} (age: {age}s)` to stderr, return file content
     - If expired: fall through to remote fetch

2. **Remote fetch with retry** (matching modules.sh lines 130-168):
   - Loop up to `MAX_RETRIES` (3) attempts
   - Each attempt: `reqwest::blocking::get(url)` with connect timeout 10s
   - On success: read body, check non-empty, print `done ({size} bytes)` to stderr, break
   - On failure: print `failed` to stderr, if not last attempt print `Retrying ({n+1}/3)...`, sleep 2 seconds
   - If all attempts fail: return `anyhow::Error` with URL in message

3. **SHA256 verification** (matching modules.sh lines 170-188):
   - Call `fetch_manifest(config)` to get manifest content
   - If manifest available:
     - Parse: find line ending with `  {action_id}.sh`, extract first field (64-char hex hash)
     - Compute: `sha2::Sha256::digest(content_bytes)` → hex encode
     - Compare: if mismatch → return error "Checksum mismatch for {action_id}.sh — possible tampering or corruption"
     - If match: print `[verified] SHA256 checksum OK` to stderr
   - If manifest unavailable: print `[WARN] Cannot verify checksum — manifest unavailable` to stderr (soft-fail)

4. **Cache store** (matching modules.sh lines 190-196):
   - `std::fs::create_dir_all(&config.cache_dir)` (ignore errors)
   - Write content to `{cache_dir}/.tmp_{action_id}` (atomic write pattern)
   - Rename `.tmp_{action_id}` → `{action_id}` (atomic on same filesystem)
   - If rename fails, try direct write as fallback

5. Return the script content as `String`

**Helper function `verify_sha256(content: &str, expected_hex: &str) -> bool`:**
- Use `sha2::Sha256` from the `sha2` crate
- Compute digest of content bytes
- Compare hex-encoded digest with expected_hex (case-insensitive)

**Progress reporting:**
- Print download progress to stderr: `Downloading {action_id}.sh... done ({size} bytes)`
- Match modules.sh stderr output format exactly
- Use `eprintln!` for all status messages

**Important implementation notes:**
- Use `reqwest::blocking::Client` with connect_timeout for all HTTP requests
- Do NOT handle community/* action_ids — return error if action_id starts with "community/"
- Cache file names use action_id directly (e.g., "install_go" → `{cache_dir}/install_go`)
- All error messages should include the action_id and URL for debugging

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub raw URLs → fetch.rs | Untrusted script content downloaded from the internet |
| fetch.rs → cache directory | Writing untrusted content to local filesystem |
| fetch.rs → execute.rs (Phase 18-02) | Passing fetched script content to execution engine |
| FLU_MODULES_BASE_URL env var | User-controlled override of module source URL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Tampering | fetch_module | mitigate | SHA256 verification against MANIFEST.sha256 before returning content. Mismatch blocks execution with error. |
| T-18-02 | Spoofing | resolve_url | mitigate | Default base URL is hardcoded to official GitHub repo. Override via FLU_MODULES_BASE_URL is user-initiated only. |
| T-18-03 | Information Disclosure | cache_dir | accept | Cache contains only public module scripts. No secrets in cache. Directory permissions follow umask. |
| T-18-04 | Denial of Service | fetch_module | mitigate | 3-attempt retry limit with 2s delay. Connect timeout 10s. Total worst case: ~36s then error. |
| T-18-05 | Elevation of Privilege | cache file write | mitigate | Cache file names derived from action_id (alphanumeric + underscore only from menu.db). No path traversal possible since action_ids are validated by menu system. |
| T-18-06 | Tampering | cache store | mitigate | Atomic write pattern (write to .tmp then rename). SHA256 verified before cache store, so cached content is pre-verified. |
</threat_model>
