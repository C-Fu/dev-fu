### Section 12: `flu_registry_fetch()` — Fetch and cache registry index

**Logic:**
1. Check for cached registry at `$FLU_CACHE_DIR/registry.json`.
2. If cached file exists AND age < FLU_CACHE_TTL: use cached version (same TTL pattern as module cache).
3. If expired or missing: fetch from `FLU_REGISTRY_URL` using curl/wget (same pattern as `_flu_fetch_manifest`).
4. After fetching official registry, check `FLU_REGISTRIES` env var (space-separated URLs).
5. For each additional URL in FLU_REGISTRIES: fetch that registry JSON too.
6. Merge all registries: official first, then extras in order. Since each is a JSON array, concatenate them.
7. Write merged result to `$FLU_CACHE_DIR/registry.json`.
8. Print merged JSON to stdout.
9. Return 0 on success (even if some extra registries fail), 1 if official registry fails AND no cache exists.

**JSON parsing approach:** Since this is a pure POSIX sh project with zero dependencies, use awk for simple JSON parsing. The registry JSON is a flat array of flat objects (no nesting), so awk can handle it. Do NOT require jq.

**IMPORTANT:** If FLU_REGISTRIES is empty, just fetch the official registry. If a third-party registry URL fails, log a warning to stderr and continue (soft-fail per registry resilience).

### Section 13: `flu_registry_lookup <action_id>` — Look up a module in registry

**Parameters:**
- `$1` = action_id WITHOUT the `community/` prefix (e.g., "install_neovim")

**Logic:**
1. Call `flu_registry_fetch` to get the merged registry (uses cache when available).
2. Parse the JSON with awk to find the entry where `"action_id"` matches `$1`.
3. Extract and set globals:
   - `_freg_name` — module display name
   - `_freg_description` — module description
   - `_freg_category` — category for menu grouping
   - `_freg_platforms` — supported platforms
   - `_freg_base_url` — base URL for fetching the script
   - `_freg_sha256` — expected SHA256 checksum
4. Return 0 if found, 1 if not found.

**Awk parsing pattern for flat JSON objects:**
```sh
_flu_reg_json | awk -v id="$1" '
  /"action_id"/ { current_id = $0; gsub(/.*"action_id": *"/, "", current_id); gsub(/".*/, "", current_id) }
  current_id == id && /"name"/ { ... extract name ... }
  ...
'
```

### Section 14: `flu_registry_fetch_module <action_id>` — Fetch a community module

**Logic:**
1. Call `flu_registry_lookup "$action_id"` to get base_url and sha256.
2. Build full URL: `${_freg_base_url}${action_id}.sh`
3. Fetch using curl/wget (same retry pattern as `flu_module_fetch`).
4. Verify SHA256 against `_freg_sha256` (same pattern as official module verification).
5. If SHA256 mismatch: print error, return 1 (per D-12 — same security model).
6. Cache to `$FLU_CACHE_DIR/community_${action_id}` (different cache key than official modules).
7. Print script content to stdout.
8. Return 0 on success.

### Modify `flu_module_resolve_url` to handle community modules:

In the existing `flu_module_resolve_url()` function (line 45-51), add a check at the beginning:
```sh
flu_module_resolve_url() {
  _fmr_action="$1"
  # Community modules — delegate to registry
  case "$_fmr_action" in
    community/*)
      # Strip community/ prefix for registry lookup
      _fmr_reg_id="${_fmr_action#community/}"
      flu_registry_lookup "$_fmr_reg_id" || return 1
      printf '%s%s.sh\n' "$_freg_base_url" "$_fmr_reg_id"
      unset _fmr_action _fmr_reg_id
      return 0
      ;;
  esac
  # Official modules — existing logic
  _fmr_base="${FLU_MODULES_BASE_URL:-https://raw.githubusercontent.com/C-Fu/dev-fu/flu.sh/modules/}"
  printf '%s%s.sh\n' "$_fmr_base" "$_fmr_action"
  unset _fmr_action _fmr_base
  return 0
}
```

This means the existing `flu_module_fetch()` and `flu_batch_run()` will automatically work with community modules because they call `flu_module_resolve_url()` for URL resolution. In menu.sh, find the function that loads menu.db (likely `flu_menu_load` or similar). Add a mechanism to inject community module entries.

**Approach:** Rather than modifying menu.sh's internals (risky — complex TUI code), generate a temporary merged menu file in flu.sh that combines menu.db + community entries. This is cleaner and doesn't touch the TUI rendering engine.

In flu.sh, AFTER the registry pre-fetch and BEFORE the main event loop, add:

```sh
# ──────────────
# 🔀 Dynamic Menu Assembly
# ──────────────
# Build merged menu: official menu.db + community modules from registry
FLU_MENU_FILE_MERGED="/tmp/flu_menu_merged_$$.db"

# Start with official menu
cat "$FLU_SCRIPT_DIR/menu.db" > "$FLU_MENU_FILE_MERGED"

# Append community modules if registry was fetched
if [ -f "$_flu_registry_cache" ] && [ -s "$_flu_registry_cache" ]; then
    _flu_cm_count=$(awk '
        /"action_id"/ {
            gsub(/.*"action_id": *"/, ""); gsub(/".*/, "")
            id = $0
        }
        /"name"/ {
            gsub(/.*"name": *"/, ""); gsub(/".*/, "")
            name = $0
        }
        /"category"/ {
            gsub(/.*"category": *"/, ""); gsub(/".*/, "")
            cat = $0
        }
        /\}/ && id != "" {
            printf "Community Modules|%s|%s|community/%s\n", cat, name, id
            id = ""; name = ""; cat = ""
        }
    ' "$_flu_registry_cache" 2>/dev/null)
    
    if [ -n "$_flu_cm_count" ]; then
        printf '\n# ── 🌐 Community Modules (from registry) ──\n' >> "$FLU_MENU_FILE_MERGED"
        printf '%s\n' "$_flu_cm_count" >> "$FLU_MENU_FILE_MERGED"
    fi
fi

# Point menu to merged file
FLU_MENU_FILE="$FLU_MENU_FILE_MERGED"
```
