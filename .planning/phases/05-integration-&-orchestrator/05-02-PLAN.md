**Section 2: TTY reattachment (per D-02, satisfies ENGN-09)**

Follow the fu.sh pattern (lines 41-47) but with the D-02 fallback behavior — do NOT exit when /dev/tty is unavailable, instead fall through to let tui.sh's `_tui_check_tty` set `_tui_use_tui=false` for numbered prompt mode:

```sh
# ──────────────
# 📡 TTY Reattachment (for curl | bash)
# ──────────────
# If stdin is not a TTY (e.g., curl-pipe-bash), reattach to /dev/tty
# so interactive TUI menus work correctly.
# If /dev/tty is unavailable, fall through to numbered prompt fallback.
if [ ! -t 0 ] && [ -r /dev/tty ]; then
    exec 0</dev/tty
fi
```

If stdin is a pipe AND /dev/tty is readable: `exec 0</dev/tty` (matches fu.sh).
If stdin is a pipe AND /dev/tty is NOT readable: do nothing — tui.sh will detect `[ ! -t 0 ]` and set `_tui_use_tui=false`.
If stdin IS a TTY (normal execution): do nothing — tui.sh detects it and uses TUI mode.

**Section 7: Main Event Loop (per D-04, D-06)**

```sh
# ──────────────
# 🔄 Main Event Loop
# ──────────────

# Track module execution for flow control
_flu_running=true

while [ "$_flu_running" = "true" ]; do
    # --- Step 1: Menu Navigation ---
    # flu_menu_navigate() handles its own TUI lifecycle:
    #   - Calls tui_init() internally
    #   - Renders menu levels, handles keyboard input
    #   - Calls tui_restore() on exit (cancel at root) or before returning (leaf select)
    # Returns 0 on leaf selection (TUI_RESULT set to "L1|L2|L3" path)
    # Returns 1 on cancel at root level
    flu_menu_navigate "$FLU_MENU_FILE"
    _flu_nav_rc=$?

    if [ "$_flu_nav_rc" -ne 0 ]; then
        # User cancelled at root — exit cleanly
        _flu_running=false
        continue
    fi

    # --- Step 2: Extract Action ID ---
    # TUI_RESULT is set by flu_menu_navigate on leaf selection
    # Example: "Developer Tools|Languages|Python"
    # flu_menu_get_action extracts the action field from menu.db
    _flu_action=$(flu_menu_get_action "$TUI_RESULT")

    if [ -z "$_flu_action" ]; then
        # No action defined for this path — edge case, return to menu
        continue
    fi

    # --- Step 3: Module Execution with Spinner (INTG-01, D-05) ---
    # Start the spinner BEFORE flu_module_execute so it's visible
    # during the network fetch phase (flu_module_fetch uses curl/wget).
    # The spinner renders via background process.
    # flu_module_execute internally:
    #   1. flu_module_fetch() — network call (spinner visible here ✓)
    #   2. flu_module_parse_metadata()
    #   3. flu_module_set_env()
    #   4. Platform compatibility check
    #   5. flu_module_collect_params() — TUI prompts (may clear screen)
    #   6. Execute module in subshell (spinner may briefly show)
    #   7. flu_module_display_result() — TUI modal (clears screen, covers spinner)
    # After flu_module_display_result, tui_restore() is called internally.
    flu_spinner_start
    flu_module_execute "$_flu_action"
    _flu_mod_rc=$?
    flu_spinner_stop

    # --- Step 4: Post-Execution ---
    # Clear screen to prepare for fresh menu render on next iteration
    clear_screen

    # flu_module_display_result already showed the result modal.
    # _flu_mod_rc is preserved but not used here — error recovery
    # is handled in Plan 05-03 (error mapping).
done

# --- Step 5: Clean Exit ---
tui_restore
printf '%sflu.sh — Goodbye!%s\n' "$TUI_GREEN" "$TUI_RESET"

# Cleanup globals
unset _flu_action _flu_nav_rc _flu_mod_rc _flu_running
unset FLU_SCRIPT_DIR FLU_MENU_FILE
```

**Flow details and edge cases:**

1. **flu_menu_navigate return values:**
   - Returns 0 + sets TUI_RESULT to "L1|L2|L3" path (leaf selected)
   - Returns 1 + TUI_RESULT='' (user cancelled at root level — Esc or q from main menu)
   - After flu_menu_navigate returns with leaf selection, TUI is in restored state (navigate calls tui_restore before returning on leaf select — verify by reading menu.sh lines 558-567 for the Enter key leaf selection path)

2. **flu_menu_get_action usage:**
   - Takes TUI_RESULT (the path string) as argument
   - Returns the action field from menu.db (e.g., "install_python")
   - Empty result means no action defined — `continue` to return to menu

3. **Spinner lifecycle during module execution:**
   - `flu_spinner_start()` launches background process, stores PID
   - `flu_module_execute()` is a blocking call (several seconds for fetch + execution)
   - During `flu_module_fetch()` (curl/wget), spinner renders on screen
   - When `flu_module_display_result()` calls `tui_init()` + `clear_screen()`, the spinner output is overwritten by the modal — this is acceptable, the fetch phase is complete
   - `flu_spinner_stop()` after module execution kills the background process, clears spinner area
   - If flu_module_execute returns early (fetch failure, cancellation), spinner is still stopped

4. **ShellCheck pragmas:**
   - Add `# shellcheck disable=SC2154` before TUI_RESULT usage (set by sourced function)
   - Add `# shellcheck disable=SC2086` if any variable expansion needs word splitting

**Convention requirements:**
- All internal loop variables prefixed with `_flu_` to distinguish from FLU_* globals
- Clean up all local variables with `unset` before exit
- Use `[ "$var" = "value" ]` not `[[ ]]` (bashism)

## STRIDE Threat Register

|-----------|----------|-----------|-------------|-----------------|
| T-05-03 | S | TTY reattachment (exec 0</dev/tty) | accept | Redirecting stdin from /dev/tty is standard practice for curl-pipe-bash scripts. /dev/tty is the controlling terminal — no spoofing risk. |

| T-05-05 | E | `flu.sh` script itself — deployed via curl-pipe-bash | accept | This is the deployment model. Users should verify the source (GitHub) before piping to sh. Same trust model as fu.sh and most curl-pipe-bash tools. |
