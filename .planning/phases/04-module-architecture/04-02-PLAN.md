Set all 7 environment variables. Adapt fu.sh patterns for POSIX sh (no `local`, no `[[ ]]`):

```sh
flu_module_set_env() {
  # FLU_OS: detect via uname
  _fse_os=$(uname -s | tr '[:upper:]' '[:lower:]')
  case "$_fse_os" in
    linux*)  FLU_OS="linux" ;;
    darwin*) FLU_OS="darwin" ;;
    *)       FLU_OS="linux" ;;
  esac

  # FLU_DISTRO: parse /etc/os-release
  if [ -f /etc/os-release ]; then
    FLU_DISTRO=$( (. /etc/os-release 2>/dev/null && printf '%s' "${ID:-linux}") )
  else
    FLU_DISTRO="linux"
  fi

  # FLU_PKG_MGR: detect available package manager
  FLU_PKG_MGR=$(_flu_detect_pkg_mgr)

  # FLU_ARCH: CPU architecture
  FLU_ARCH=$(uname -m)

  # FLU_IS_WSL: check /proc/version for Microsoft
  if grep -qi "microsoft" /proc/version 2>/dev/null; then
    FLU_IS_WSL="1"
  else
    FLU_IS_WSL="0"
  fi

  # FLU_IS_TERMUX: check env var and directory
  if [ -n "${TERMUX_VERSION:-}" ] || [ -d /data/data/com.termux ]; then
    FLU_IS_TERMUX="1"
  else
    FLU_IS_TERMUX="0"
  fi

  # FLU_IS_ROOT: check effective UID
  if [ "$(id -u)" -eq 0 ]; then
    FLU_IS_ROOT="1"
  else
    FLU_IS_ROOT="0"
  fi

  # Export all vars to module environment
  export FLU_OS FLU_DISTRO FLU_PKG_MGR FLU_ARCH
  export FLU_IS_WSL FLU_IS_TERMUX FLU_IS_ROOT

  unset _fse_os
}
```

**Part B: Implement `flu_module_collect_params(param_string)` (per D-10):**

- Accept param_string as $1 (the `@params` value from metadata, e.g., "scope=radio:global,user;name=text")

- Read parsed lines one by one using `while IFS='|' read -r _idx _name _type _choices`
- For each line, dispatch to the appropriate Phase 2 widget based on type:
  - `_type = "radio"` → Call `tui_radio "$_name" "Choose $_name"` with the comma-separated choices split into positional args. Map the returned 0-based index back to the choice string. If tui_radio returns non-zero (user cancelled), set `_flu_cancelled=true` and break.
  - `_type = "text"` → Call `tui_text_input "$_name" "Enter $_name"`. After the call, use TUI_RESULT as the value. If TUI_RESULT is empty (user cancelled via Esc), set `_flu_cancelled=true` and break.
  - `_type = "yesno"` → Call `tui_yesno "$_name" "Enable $_name?" "no"`. After the call, use TUI_RESULT ('yes'/'no'). If returns non-zero (user cancelled), set `_flu_cancelled=true` and break.
- For each collected value, construct a `--key value` arg pair and append to `_flu_module_args`

  - text: `_flu_module_args="${_flu_module_args} --${_name} ${TUI_RESULT}"`
  - yesno: `_flu_module_args="${_flu_module_args} --${_name} ${TUI_RESULT}"`
- If `_flu_cancelled=true`, return 1 (abort execution)
- Otherwise, set global `_flu_module_args` for the executor to use, return 0

**Part A: Implement `_flu_execute_with_timeout(timeout_sec, url, ...)` (per D-15):**

- Accept timeout_sec as $1, url as $2, shift 2, remaining args are module args
- Try `timeout` command first: `if command -v timeout >/dev/null 2>&1; then timeout "$_timeout" sh -s -- "$@" ... fi`
- Fallback (no `timeout` command): start background subshell, sleep for timeout then kill
  ```sh
  # POSIX sh background process + kill pattern:
  (
    set -euo pipefail
    curl -fsSL "$url" | sh -s -- "$@"
  ) &
  _pid=$!
  (
    sleep "$_timeout"
    kill "$_pid" 2>/dev/null || true
  ) &
  _watchdog=$!
  wait "$_pid" 2>/dev/null
  _rc=$?
  kill "$_watchdog" 2>/dev/null || true
  wait "$_watchdog" 2>/dev/null || true
  ```
- Return the exit code from the module execution (or signal code if killed by timeout)
- On timeout kill: return 124 (same as `timeout` command convention)

**Part B: Implement `flu_module_execute(action_id)` (per D-09, D-14, D-16):**

This is the main orchestrator. Execution order (D-09):

1. **Fetch:** Call `flu_module_fetch "$action_id"` and capture output to a variable/pipe. To capture in POSIX sh without `$()`, use a temp approach: `flu_module_fetch "$action_id" > "$_flu_tmp_script"` or pipe directly. Since D-05 says pipe-to-sh, pipe through metadata parse first.

   Actually: Since we need to parse metadata AND execute, and D-05 says no temp file, we need to fetch twice OR fetch once and use a pipe with tee. Better approach: fetch into a variable using a temporary file (this is an implementation detail, not user-visible). Use `_flu_tmp_script` in /tmp for the single invocation, clean up after.

   Wait — D-05 says "pipe-to-sh execution. No temp file." But we need to parse metadata before execution. Re-reading D-05: "Pipe-to-sh execution (curl-pipe-bash style). No temp file — download and pipe directly." This means the module script is fetched and piped directly, not saved. For metadata parsing, we parse from the fetched content before piping to sh.

   Implementation approach: Fetch script content → store in a variable via `_flu_script_content=$(flu_module_fetch "$action_id")`. Parse metadata from `printf '%s' "$_flu_script_content" | flu_module_parse_metadata`. Then pipe the same content to sh: `printf '%s' "$_flu_script_content" | sh -s -- $args`.

   Wait, we can't use `$()` in POSIX sh the same way? Actually we CAN use `$()` — it's POSIX. But `$()` with large scripts could be problematic. Better: Use a pipeline.

   Solution: `flu_module_fetch "$action_id" | tee >(flu_module_parse_metadata) | sh -s -- $args` — but `tee >(...)` is bashism. POSIX alternative:
   
   ```sh
   # Fetch + parse + execute in a single pipeline:
   flu_module_fetch "$action_id" | {
     # First pass: parse metadata (save to a temp file via awk)
     flu_module_parse_metadata
     # Second: the same input also goes to sh
     # ... but awk consumed the input ...
   }
   ```
   
   This is tricky in POSIX. The practical approach: use a temporary file BUT delete it immediately after. The "no temp file" rule from D-05 is about user-facing behavior — internally, using /tmp briefly is fine since the module content is transient.

   FINAL APPROACH (pragmatic, POSIX-safe):
   ```sh
   _fetmp="/tmp/flu_module_$$.sh"
   flu_module_fetch "$action_id" > "$_fetmp" || { rm -f "$_fetmp"; return 1; }
   flu_module_parse_metadata < "$_fetmp" || { rm -f "$_fetmp"; return 1; }
   # ... collect params, construct args ...
   # Execute
   sh "$_fetmp" -- $args
   _rc=$?
   rm -f "$_fetmp"
   ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input → widget → module args | Parameter values from tui_text_input are passed as --key value args to sh |
| Environment → module subshell | FLU_* env vars are exported into the module's execution context |
| Subshell → parent shell | Exit code must propagate correctly; parent must not crash on subshell failure |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | flu_module_collect_params() — text input | mitigate | tui_text_input accepts arbitrary text. Module args are shell-quoted via double-quote wrapping in `"$value"`. Module script is responsible for its own input sanitization. |
| T-04-06 | Elevation of Privilege | flu_module_execute() — subshell | mitigate | Modules run as current user (D-16). No sudo escalation by the framework. Each module handles privilege internally — same pattern as fu.sh _maybe_sudo(). |
| T-04-07 | Denial of Service | _flu_execute_with_timeout() — watchdog | mitigate | Timeout watchdog cleans up via kill. Default 300s timeout prevents runaway modules. Watchdog process itself is killed after the main process exits. |
| T-04-08 | Information Disclosure | flu_module_set_env() — env vars | accept | FLU_* vars expose system info that is already available to any process via uname, /etc/os-release. No sensitive data exposed. |
</threat_model>
